# Waysheet — План проекта (актуально на 27.04.2026)

## 1) Цель
Кроссплатформенное desktop-приложение для учёта путевых листов с понятным русскоязычным интерфейсом, встроенным управлением БД и быстрым ежедневным вводом данных.

## 3) Архитектурные ориентиры (без ломки текущего UX)
- Сохраняем pattern: `Toolbar -> Filters/Group actions -> List (left) -> Editor (right)`.
- Любое изменение данных — через автосохранение, без лишних модальных шагов.
- Любое добавление — с немедленным переходом в редактирование новой записи.
- Все текстовые и иконковые glyphs — из локальных шрифтов в каталоге проекта/сборки.

## 4) Сборка и запуск по платформам

### Windows

```powershell
cmake -S . -B build
cmake --build build --config Release
.\build\Release\waysheet.exe
```

### Arch Linux / Linux

```bash
cmake -S . -B build-linux -DCMAKE_BUILD_TYPE=Release
cmake --build build-linux --config Release
./build-linux/waysheet
```

Для первой GUI-сборки нужен доступ к сети: CMake скачивает GLFW, Dear ImGui и ImGuiFileDialog через `FetchContent`.

Установка:

```bash
cmake --install build-linux
```

Для пользовательской установки:

```bash
cmake --install build-linux --prefix "$HOME/.local"
```

Особенности Linux:

- GUI требует рабочую Wayland/X11-сессию (`WAYLAND_DISPLAY` или `DISPLAY`).
- Установленное приложение загружает шрифты из `share/aPList/fonts`, build-запуск — из локального `fonts/`.
- В sandbox/CI без графического окружения запускать GUI под виртуальным дисплеем.
- На Arch Linux нужны CMake, C++20 toolchain, SQLite, OpenGL/Mesa и Wayland/X11 зависимости.

## 5) План следующих итераций

### Итерация A — Итоги и отчеты
1. [x] Печатный HTML-отчет итогов за месяц.
2. Уточнить актуальные отчетные формы после стабилизации рабочих сценариев.

### Итерация B — Дашборд и аналитика
1. KPI за выбранный период: пробег, норматив, факт, отклонение.
2. Топ маршрутов и машин по пробегу/перерасходу.
3. Мини-графики по дням.

### Итерация C — Качество и стабильность
1. Проверки валидации дат/чисел/обязательных полей.
2. Регрессионные тест-сценарии для CRUD и автосохранения.
3. Улучшение сообщений об ошибках БД и UX при отказах операций.

## 6) Критерии готовности ближайшей рабочей версии
- Пользователь запускает приложение и сразу работает с последней БД.
- Любая запись (справочники + путевые листы) добавляется и редактируется без «пустых» кликов.
- Интерфейс русскоязычный, шрифты и иконки стабильно отображаются на чистой машине.
- Базовые отчётные операции (минимум CSV/печать) доступны из UI.
